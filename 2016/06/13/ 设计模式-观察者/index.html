写在前面

观察者模式运用广泛，且实现相对简单。如著名的 MVC模式中Model和View的交互就运用了观察者模式，还有很多框架都运用其进行事件的通知处理。

什么是观察者模式

观察者设计模式定义了对象间的一种一对多的依赖关系，以便一个对象的状态发生变化时，所有依赖于它的对象都得到通知并自动刷新。

为什么要用观察者模式

设想这样一个场景：杂志社会在每期杂志更新的时候把杂志寄到订阅该杂志的用户家里。这里的观察者是每个用户，被观察者是杂志社。一旦有新的杂志，杂志社就会“通知”每个用户，当然这里的“通知”就是以邮寄的方式寄到用户家。而这些能收到杂志的用户必须是订阅过杂志，如果用户取消订阅，则他以后再也收不到。这是观察者模式的最好应用。

如果不利用观察者模式，上面的情景可能是这样：每个用户需要每天都打电话到杂志社询问本期杂志是否更新了。这样一来，作为用户真是太累了，因为杂志一般是每个月的某一天才出，而他却需要每天去询问，确保没有错过最新杂志。而对于杂志社来说也很累，每天需要回应大量用户来电。

由此看来，观察者模式能够极大地解耦,杂志社只需在新用户订阅的时候，记下他的地址，而杂志更新的时候批量寄出去。杂志社并不需要关心用户是谁，买杂志的用途。用户所需要做的也仅仅是订阅杂志，然后就等着它更新。用户不需要关心杂志社怎么制作杂志，什么时候更新。并且这种方式对于双方都极为高效。

怎么用观察者模式

实现一个观察者模式只需要四个类：

  • Subject 主题抽象类（所有的被观察者继承于它）
  • Observer 接口（所有观察者实现该接口）
  • Magazine 被观察者具体类（杂志社）
  • User 观察者（订阅杂志的用户）

具体步骤：

  1. Subject 主题抽象类中添加三个方法最基本的方法：注册观察者，删除观察者，通知所有观察者；以及维护一个所有观察者的集合。
  2. Observer 接口 只有一个方法：更新通知。这个方法在被观察者的通知方法中被调用。
  3. Magazine 继承 Subject 抽象类，可新增一个状态改变方法，并在该方法中调用父类的通知观察者方法，以便在状态改变通知所有的观察者。
  4. User 观察者具体类，实现 Observer 接口中的 更新方法，以便在有通知时，对通知的状态进行进一步处理。

拉和推

观察者模式中的通知分为拉和推两种方式。上面的代码采用的是推的方式，即由被观察者推送给观察者，可以发现其有一定的缺陷，比如被观察者通知观察者是通过调用观察者的update方法，但却只传一个固定的参数，可能是状态字符串或者一个对象模型。这样当以后有新的事情需要通知时，就可能需要提供新的update方法。

而采用拉的方式，即观察者主动去拉取所需通知。拉的实现很简单，只需要将通知时，即调用update方法时，传入被观察者自身，并提供被观察者属性的getter方法。就能让观察者按照自己的需要去取。

Java中的观察者模式

Java 在 JDK 中也对观察者模式提供了应用支持，在java.util包下提供Observable类和Observer接口，分别为被观察者和观察者，即Observable类相当于我们的Subject类，Observer接口相当于我们的Observer接口。实现方式大致相同，只是 Java 中多了更多细节控制，且不用我们去写这两个抽象类和接口便可直接使用。
